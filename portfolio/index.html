问题 & 方案

痛点

迭代历程

v1.0 — RSS + PubMed 采集，单维度评分
搭建基础管线：feedparser + E-utilities → Article 数据模型 → LLM 评分摘要 → 飞书推送。SQLite 去重保障不重复打扰。
v1.1 — 双维度评分
将单一分数拆为学术相关性和实践影响力两个独立维度，加权合并。避免「顶刊平庸论文」和「小刊方法突破」被错误排序。
v1.2 — YouTube 播客字幕提取
接入 3 个医学 AI 播客源（NEJM AI、JAMA+、a16z）。用 yt-dlp 替代 YouTube Data API，RSS 获取视频列表，字幕 json3 解析为纯文本。
⚡ 挑战：GitHub Actions 机房 IP 被 YouTube 反爬拦截，字幕请求返回 403
v1.3 — 反爬双保险
Cloudflare WARP 伪装住宅 IP（零维护）+ cookies.txt 注入认证（兜底）。字幕提取成功率从 0% 恢复到 100%。
v2.0 — 双管线并行
播客和论文走不同 LLM prompt：论文用 PICO 框架 + 商业化探索的结构化摘要，播客用金句混音 prompt（开篇金句 → 讲者背景 → 反直觉见解 → 直接引用）。跳过分数的播客自动路由到混音管线。

关键设计决策

为什么双维度评分而非一个总分？
单维度容易将「顶级期刊的平庸论文」和「小型期刊的方法学突破」混为一谈。分离学术相关性和实践影响力，加权合并更准确。
为什么 yt-dlp 而非 YouTube Data API？
Data API 需要 API Key + OAuth 且有配额限制。yt-dlp 纯 Python 库零配额。配合 Cloudflare WARP + cookies 解决了机房 IP 反爬问题。
为什么 SQLite 而非 Redis/Postgres？
GitHub Actions runner 是无状态的，SQLite 单文件通过 git commit 持久化。174 条记录查询 <2ms，零运维，零成本。
为什么设计独立的播客混音管线？
播客文字稿 5000-8000 字，和论文摘要信息密度完全不同。独立 prompt + 独立路由，不做评分直接生成结构化简报，保持每种内容的处理质量。
